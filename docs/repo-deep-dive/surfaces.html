Integration surfaces

Telegram and Discord adapters are outside this repo

This repository contains the agent runtime and several local/embed surfaces. It does not contain Telegram, Discord, Slack, or webhook adapter code. The root README points chat automation to the separate earendil-works/pi-chat project.

How chat platforms integrate: a platform bridge should map external conversations to Pi sessions, send platform messages through SDK or RPC prompts, subscribe to events, and translate assistant/tool/status updates back into platform messages.

Available surfaces in this repo

Interactive TUI

Local human terminal surface using InteractiveMode and packages/tui. Supports rich extension UI, selectors, overlays, editor replacement, custom widgets, and terminal-specific rendering.

Print mode

Single-shot text output for simple automation. Reads prompt/pipe input and prints final assistant text.

JSON mode

Single-shot JSONL stream. Emits session header and AgentSessionEvents for custom consumers that do not need bidirectional control.

RPC mode

Headless LF-delimited JSON over stdin/stdout. Supports prompt, steer, follow-up, abort, state, model/thinking, queue modes, compaction, retry, bash, sessions, export, fork/clone, messages, commands, and extension UI requests.

Node/TS SDK

Preferred for first-party Node services. Use createAgentSession() or createAgentSessionRuntime(), subscribe to events, provide custom tools/resources/storage, and control sessions directly.

Web UI

Browser component library built on pi-agent-core and pi-ai. Useful for web apps and dashboards; it is not a bot/webhook server.

Recommended platform bridge designs

  1. Node/TypeScript bot service: use the SDK. Create one AgentSession per platform thread/channel/user mapping. Subscribe to events. Store the mapping in your service database. Translate platform messages to session.prompt(), steer(), or followUp().
  2. Non-Node or isolated process: spawn pi --mode rpc. Communicate using strict LF JSONL. Implement RPC commands and respond to extension UI requests if extensions need dialogs.
  3. Simple command automation: use pi -p for final text only or pi --mode json for one-shot streamed events.
  4. Browser app: use packages/web-ui components with pi-agent-core; add your own server/platform ingestion layer.

RPC as platform boundary

RPC is the language-agnostic surface. Commands cover prompts, steering, follow-ups, abort, state, model selection, thinking level, compaction, retry, bash, sessions, export, fork/clone, message access, and command listing. RPC also defines an extension UI sub-protocol for select/confirm/input/notify/status/widget/title/editor messages.

RPC gotchas: framing is strict LF-delimited JSONL; do not use line readers that split on Unicode separators. TUI-only extension UI features degrade or no-op in RPC.

SDK as platform boundary

The SDK is richer because it avoids subprocess protocol translation. A platform bridge can supply custom SessionManager, AuthStorage, ResourceLoader, model registry, tool allowlist, custom tools, and extension factories. It can subscribe to AgentSessionEvents directly and decide how aggressively to update external chat messages.

What to translate for chat platforms

Incoming

External user text/images/files become Pi user messages or attachment-aware prompt content. Thread/channel id selects or creates a session.

Streaming

message_update assistant deltas can edit a pending platform message. Some platforms rate-limit edits, so buffer or throttle.

Tools

tool_execution_* can become status messages, collapsed logs, or hidden telemetry depending on platform UX.

Interactive UI

TUI dialogs do not translate automatically. For RPC, implement extension UI responses; for SDK, provide your own UI abstraction or avoid TUI-only extensions.
