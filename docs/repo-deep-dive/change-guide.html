Where to change what

Decision map for future work

Use the smallest seam that matches the change. Most product features belong in packages/coding-agent; only generic loop semantics belong in packages/agent; only provider abstraction belongs in packages/ai.

Common tasks

Add a built-in coding tool

Start in packages/coding-agent/src/core/tools/. Add a definition, export it in tools/index.ts, update default sets if needed, and ensure prompt snippets/renderers/tests exist.

Add an extension tool

Use pi.registerTool() in an extension. This avoids touching core and gets schema validation, execution, hooks, and rendering integration.

Add memory behavior

Prefer an extension using custom entries plus context or before_agent_start. Touch SessionManager only if changing the session format itself.

Add a slash command

For core TUI commands, update slash-commands.ts and interactive submit routing. For reusable/project commands, prefer prompt templates, skills, or extension commands.

Add a provider

For built-ins, update packages/ai provider registration/model generation and coding-agent model/auth docs. For local/custom providers, prefer models.json or extension registerProvider().

Add platform integration

Do not wire Telegram/Discord into the TUI. Build a bridge on SDK or RPC. Map external conversation ids to Pi sessions and translate events.

Change TUI rendering

Most UI changes are in interactive-mode.ts or modes/interactive/components/. Generic renderer/focus/overlay behavior belongs in packages/tui.

Change config/resource loading

Use settings-manager.ts for settings schema/merge, resource-loader.ts for load order, and package-manager.ts for package discovery/precedence.

Do not cross these wires

  • Do not put provider-specific request logic in packages/agent. The loop must stay provider-neutral.
  • Do not put terminal-specific UI assumptions into AgentSession. Modes should consume session events.
  • Do not assume sessions are flat logs. They are branch trees.
  • Do not make private extension state model-visible accidentally. custom is private; custom_message is model-visible.
  • Do not bypass resource precedence unless a CLI override explicitly requires it.
