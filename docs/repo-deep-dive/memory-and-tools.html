Deep integration

Memory and tools are session-shaped

Pi does not have a single magic memory database abstraction. Durable memory is primarily append-only session JSONL plus extension-controlled custom entries/messages. Tools are provider-neutral schemas wrapped into agent-executable definitions and rendered by surface-specific renderers.

Memory model: what exists

In-memory agent state

Agent.state.messages is the live transcript used by the low-level loop. It changes during streaming and stores finalized messages.

Durable sessions

SessionManager persists JSONL entries. Sessions are trees, not flat logs: each entry has an id and parent id.

Model-visible memory

Messages, custom messages, branch summaries, compaction summaries, system prompt content, skills, context files, and context-hook injections can reach the model.

Private extension state

custom session entries persist branch-aware extension data but are explicitly ignored by context building unless extension code reads and injects them.

Session JSONL entries

The session file begins with a header containing type/version/id/timestamp/cwd and optional parent session. Entries are appended as JSON lines. Important entry families:

  • message: normal user, assistant, or tool-result messages.
  • model_change and thinking_level_change: restore active runtime settings along a branch.
  • compaction: stores summary and first-kept entry id; context reconstruction respects compaction boundaries.
  • branch_summary: generated when navigating branches; can participate in context.
  • custom: private extension state, not automatically sent to the model.
  • custom_message: extension-injected durable message that participates in model context.
  • label and session_info: metadata.

Branch-correct reconstruction

buildSessionContext() walks from the current leaf back to the root. It reconstructs messages and the latest model/thinking settings for the selected branch, handles compaction, includes model-visible entries, and skips private custom entries. This is the key rule for adding your own memory system: read the current branch, not the whole session file.

Memory recipe: store raw/private memory as versioned custom entries; reconstruct effective state from ctx.sessionManager.getBranch(); inject only selected summaries/facts through custom_message, before_agent_start, context, or system prompt fragments.

Memory injection points

System prompt

buildSystemPrompt() combines custom system prompt, tool snippets/guidelines, project context files, skills, date, cwd, and append prompts. Use this for stable global/project context.

before_agent_start

Extension hook before the agent starts. Can modify the system prompt for the turn or return custom messages to prepend to the prompt batch.

context

Runs before every provider call through agent.transformContext. Best for pruning, compacting, retrieving, or injecting memory at the message-list level.

custom_message

Durable model-visible extension message. Useful for memory summaries that should replay with the branch.

Tool result details

Use details for state naturally tied to a tool call. It persists and branches with that tool result, but is not itself model-visible content.

Compaction

Compaction converts old transcript into a summary and keeps selected later entries. Memory systems should decide whether they inject before or after compaction summaries.

Adding your own memory system

  1. Choose private vs model-visible state. Private facts go to custom. Model-visible summaries go to custom_message or hook injection.
  2. Version your entries. Use a stable custom type such as my-memory.v1 and data with explicit schema/version.
  3. Reconstruct from branch. On session_start and tree/session changes, scan ctx.sessionManager.getBranch() for your entries. Ignore unrelated branches unless you intentionally merge.
  4. Maintain an in-memory index. Build whatever vector/BM25/summary cache you need from branch entries or external storage, keyed by session id and branch leaf.
  5. Inject selectively. Use context for per-call retrieval or before_agent_start for per-prompt memory. Keep injected text concise and provenance-aware.
  6. Persist decisions. When memory changes, append a new custom entry. Do not mutate old JSONL entries.
  7. Handle compaction. Decide whether compaction should summarize your memory into custom_message, leave private entries alone, or prune external indexes.

Tool model: from schema to execution

At the LLM boundary, a tool is { name, description, parameters }. Coding-agent extends that with labels, execution, prompt snippets, renderers, argument preparation, and sequential/parallel hints.

  1. Definition. Built-ins and extensions create ToolDefinitions with TypeBox/JSON Schema parameters, descriptions, execution, renderers, and prompt metadata.
  2. Wrapping. wrapToolDefinition() adapts definitions into AgentTools, injecting an extension context into execution.
  3. Activation. AgentSession._refreshToolRegistry() merges built-in definitions, extension tools, SDK custom tools, allowlists, and active tool selection.
  4. Prompt visibility. buildSystemPrompt() lists selected tool names and prompt snippets/guidelines. Tools are also passed structurally in Context.tools.
  5. Provider serialization. Providers map Context.tools to their function/tool schema formats and parse streamed tool calls back into Pi toolCall blocks.
  6. Validation. The agent loop finds the tool by name, applies prepareArguments, validates args through validateToolArguments(), and converts failures into error tool results.
  7. Interception. beforeToolCall/tool_call can block. afterToolCall/tool_result can patch content/details/error/terminate.
  8. Execution. execute(toolCallId, args, signal, onUpdate) returns { content, details, terminate? }. onUpdate streams partial UI state.
  9. Result. The loop creates a ToolResultMessage: content for the model, details for UI/state/export, isError for failure.
  10. Rendering. TUI uses ToolExecutionComponent with built-in/extension renderers. HTML export reuses renderers and converts terminal output to HTML. Web UI has a separate Lit renderer registry.

Adding an extension tool

pi.registerTool({
  name: "my_tool",
  label: "My tool",
  description: "Short model-facing description.",
  parameters: Type.Object({ query: Type.String() }),
  promptSnippet: "my_tool: search the private index",
  async execute(toolCallId, params, signal, onUpdate, ctx) {
    onUpdate?.({ content: [{ type: "text", text: "Searching..." }], details: { phase: "search" } });
    const result = await searchIndex(params.query, signal);
    return {
      content: [{ type: "text", text: result.summary }],
      details: { matches: result.matches }
    };
  },
  renderResult(args, result, context) {
    return [`Found ${result.details.matches.length} matches`];
  }
});

Adding a built-in tool

  1. Create packages/coding-agent/src/core/tools/<name>.ts with a focused createXToolDefinition(cwd, options).
  2. Use TypeBox parameters and return concise, truncation-safe model content.
  3. Put structured UI/export state in details.
  4. Add promptSnippet and promptGuidelines if the model needs behavioral guidance.
  5. Add renderCall/renderResult for TUI display when raw text is not enough.
  6. Use withFileMutationQueue() for file-mutating tools so edit/write-like operations serialize safely.
  7. Export from tools/index.ts, extend ToolName, factory switches, and default tool sets if appropriate.
  8. Add tests for validation, execution, rendering, active-tool registry, and prompt snippets.
