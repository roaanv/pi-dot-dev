Terminal surface

TUI is a consumer of session events

InteractiveMode does not run the agent loop directly. It owns terminal input and rendering, delegates work to AgentSession, and maps AgentSessionEvents into TUI components.

Input-to-render flow

Interactive mode lifecycle

  1. main.ts selects interactive mode and creates AgentSessionRuntime.
  2. new InteractiveMode(runtime) creates a TUI(new ProcessTerminal()), root containers, editor, footer, and theme state.
  3. init() adds header, chat, pending message, status, extension widget, editor, and footer containers in visual order.
  4. The UI starts raw terminal input, focuses the editor, loads keybindings, binds extensions, and subscribes to the current session.
  5. run() loops forever: wait for getUserInput(), then call session.prompt(input).
  6. When sessions are replaced, runtime callbacks let the UI invalidate extension UI, unsubscribe old listeners, bind the new session, and rerender existing context.

Terminal and TUI package

packages/tui is a generic line-oriented terminal renderer. ProcessTerminal enables raw mode, bracketed paste, resize handling, keyboard protocol negotiation, and forwards normalized key sequences. TUI.handleInput() runs global input listeners, handles terminal responses, manages overlays/focus, forwards input to the focused component, and requests a render.

Rendering constraint: every component line must fit inside the width passed to render(width), except special image lines. Custom TUI components must invalidate themselves and request a render after state changes.

Editor submit routing

CustomEditor wraps the base TUI editor with app-level shortcuts. On submit, InteractiveMode.setupEditorSubmitHandler() routes input:

  • Built-in slash commands such as /settings, /model, /export, /session, /compact, /reload, /login, /resume, and /quit are handled in the UI layer.
  • Extension commands are delegated through the extension runner.
  • ! / !! run bash through the session bash execution path.
  • During streaming, normal text becomes session.prompt(text, { streamingBehavior: "steer" }) and pending UI is updated.
  • When idle, normal chat text resolves the pending getUserInput() promise; the main run loop then calls session.prompt().

Session events to components

Assistant streaming

message_start creates an AssistantMessageComponent. message_update mutates it with text/thinking/tool-call deltas. message_end finalizes aborted/error state and clears streaming references.

Tool status

Assistant tool-call content creates pending ToolExecutionComponents. tool_execution_start/update/end marks args, execution state, partial output, final result, and image conversion.

Queues and lifecycle

queue_update refreshes pending steer/follow-up display. agent_start starts loaders/progress. agent_end clears progress, orphan pending tools, and shutdown state.

Extension UI in interactive mode

Interactive mode supplies the real ExtensionUIContext: dialogs, confirmations, input prompts, notifications, terminal input listeners, status text, working indicators, widgets above/below editor, custom footer/header/title, editor control, autocomplete wrapping, overlays, and focused custom components. RPC and print cannot provide all of these; see surfaces.
