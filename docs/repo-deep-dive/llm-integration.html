Provider boundary

The agent integrates with LLMs through a stream function

The agent loop does not know OpenAI, Anthropic, Google, or Bedrock internals. It knows Model, Context, tools, options, and the assistant event stream contract.

Request path

User/session message
  -> AgentSession
  -> Agent.prompt()
  -> runAgentLoop()
  -> transformContext + convertToLlm
  -> streamFn(model, Context, SimpleStreamOptions)
  -> ModelRegistry.getApiKeyAndHeaders(model)
  -> pi-ai streamSimple(model, context, options)
  -> api-registry getApiProvider(model.api)
  -> built-in or dynamic provider streamSimple
  -> AssistantMessageEventStream
  -> AgentEvent / AgentSessionEvent / UI or RPC

Boundary contracts

Model

Contains provider identity, API identity, base URL, capabilities, costs, context window, max tokens, and metadata. model.api selects the stream implementation; model.provider drives auth/display/selection.

Context

{ systemPrompt, messages, tools }. Built just before each LLM call after context transforms and custom message conversion.

SimpleStreamOptions

Carries auth, headers, transport, session id, retry settings, payload/response hooks, reasoning level, thinking budgets, and abort signal.

Stream protocol

Providers emit start/text/thinking/toolcall deltas and terminal done/error events. Runtime failures should be encoded in the stream, not thrown past the loop.

Model registry and selection

ModelRegistry loads built-in generated models plus models.json custom models/overrides. It can register dynamic providers from extensions, reset built-ins during refresh, apply OAuth metadata, and compute available models by auth status.

  • findInitialModel() prioritizes CLI model/provider, scoped models, saved defaults, provider default models, then any available model.
  • CLI resolution searches all known models, not only auth-available ones, so first-time --api-key use can resolve a model before stored auth exists.
  • Thinking levels can be specified with suffixes and are clamped to model-supported levels.

Auth resolution

  1. AuthStorage checks runtime API key overrides.
  2. Then stored API keys in auth.json.
  3. Then stored OAuth credentials, refreshing if expired.
  4. Then environment variables or ambient sentinels reported by pi-ai env helpers.
  5. Then optional fallback resolvers.
  6. ModelRegistry.getApiKeyAndHeaders() merges provider/model headers, provider configured API keys, built-in headers, and optional authHeader behavior.

Provider dispatch

packages/ai/src/stream.ts imports built-in registration as a side effect. streamSimple() looks up model.api in the API registry and calls the provider's streamSimple. Built-in provider modules are lazy-loaded wrappers, so provider code is only imported on first use. Dynamic providers use the same registry and can be unregistered by source id.

Do not document provider internals as agent architecture. The integration point is the registry and event stream contract. Individual request payload conversion is provider-local.

Provider hooks and custom providers

  • before_provider_request lets extensions inspect/replace serialized provider payloads.
  • after_provider_response lets extensions inspect response metadata.
  • registerProvider() can add models, base URLs, API keys, headers, OAuth login, authHeader, or a custom streamSimple.
  • Provider registrations during extension load are queued until ModelRegistry exists; later registrations apply immediately and refresh session model references.

Invariants

Agent is provider-agnostic

It has no OpenAI/Anthropic/etc. branches. It only consumes pi-ai abstractions.

Coding-agent owns auth richness

The product stream function resolves API keys and headers through ModelRegistry before calling streamSimple.

Tools cross the same boundary

Context.tools is provider-neutral. Providers serialize tool schemas and parse tool-call deltas back into Pi's assistant event protocol.
